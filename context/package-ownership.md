# Package ownership mechanics

How the one-vendor-one-owner rule is actually enforced: one registry, six generated ESLint rules,
two supporting rules, and an independent script. The rationale is
[ADR 0004](../architecture/adrs/0004-package-ownership.md); the vendor-by-vendor list is
[dependency-map](./dependency-map.md).

## The registry

`eslint/package-ownership.config.mjs` exports two values and one lookup:

```js
export const FOUNDATIONAL_VENDORS = ['react', 'react-dom'];

export const PACKAGE_OWNERSHIP = [
  { vendor: 'axios', owner: '@/packages/http', ownerDirs: ['src/packages/http'] },
  // â€¦ 36 entries total
];

export function findOwnershipEntry(packageName) { â€¦ }
```

`ownerDirs` is an array because a few vendors legitimately have more than one owning directory:
`@ionic/react` is used by both the Ionic and router owners, and `@capacitor/core` by both
`src/platform/runtime` and `src/packages/secure-storage`.

Four consumers read this file and none of them restate it: the architecture ESLint plugin, the
ownership script, `.ai/graphs/ownership.json` (generated by `scripts/knowledge/lib/generate.mjs`),
and this documentation. That is the point â€” the graph an agent reads cannot disagree with the rule
that fails the build.

## The rule family

`eslint/architecture-plugin/shared/ownership-rule.factory.mjs` exports `createVendorOwnershipRule`,
which produces six rules. They differ only in `vendorFilter`:

| Rule                                                                                                 | Filter                                     |
| ---------------------------------------------------------------------------------------------------- | ------------------------------------------ |
| [no-raw-package-imports](../docs/eslint/no-raw-package-imports.md)                                   | every vendor (the catch-all)               |
| [no-raw-capacitor-imports](../docs/eslint/no-raw-capacitor-imports.md)                               | `@capacitor/*` + the secure-storage plugin |
| [no-direct-axios-import-outside-owner](../docs/eslint/no-direct-axios-import-outside-owner.md)       | `axios`                                    |
| [no-direct-ionic-import-outside-owner](../docs/eslint/no-direct-ionic-import-outside-owner.md)       | `@ionic/react`, `@ionic/react-router`      |
| [no-direct-dayjs-import-outside-owner](../docs/eslint/no-direct-dayjs-import-outside-owner.md)       | `dayjs`                                    |
| [no-direct-virtuoso-import-outside-owner](../docs/eslint/no-direct-virtuoso-import-outside-owner.md) | `react-virtuoso`                           |

The catch-all already covers the five targeted rules. The targeted rules exist for the error
message: a developer who imports `axios` in a service gets "Axios is imported only by the HTTP
owner" rather than a generic ownership complaint. Cheap redundancy, better diagnostics.

Each produced rule reports two conditions:

- **`outsideOwner`** â€” the vendor has an owner and this file is not inside it.
- **`unregisteredVendor`** â€” the vendor is in `node_modules` but absent from the registry. A new
  dependency therefore fails lint until someone decides who owns it.

Two more rules complete the boundary:

- [no-module-imports-in-package-owners](../docs/eslint/no-module-imports-in-package-owners.md) â€”
  owners wrap vendors; importing `modules`, `shared`, `platform`, or `app` would couple the owner to
  the app it is meant to insulate.
- `no-direct-navigation-outside-router-owner`
  ([doc](../docs/eslint/no-direct-navigation-outside-router-owner.md)) â€” `history.pushState` and
  `location.assign` are ownership violations that are not imports, so they are matched on the AST
  instead.

## The type-only exception

Type-only imports are erased at build time and cannot create a runtime edge, so
`import type { AxiosAdapter } from 'axios'` stays legal inside `packages/` and `platform/`. That is
what lets `src/packages/http/interfaces/http.interfaces.ts` reference the adapter type it accepts.
The exception is narrow on purpose: it does not apply to `modules/`, `shared/`, or `app/`, where
`architecture/no-raw-vendor-types-in-domain` keeps vendor types out of the domain.

The ownership script applies the same exception, more strictly: `importsVendorTypeOnly` returns true
only when **every** import of that vendor in the file is type-only. One value import taints it.

## The script

`npm run quality:package-ownership` â†’ `scripts/architecture/validate-package-ownership.mjs` proves
three things ESLint alone cannot:

1. **Registry completeness** â€” every entry in `package.json` `dependencies` is foundational or
   registered. ESLint only sees files that exist; this sees the manifest.
2. **Owner-dir existence** â€” every `ownerDirs` path is real, so a deleted owner is caught rather
   than silently disabling its rule.
3. **An independent scan** â€” it re-derives imports across `src/**` with its own regex scan rather
   than the plugin's AST pass, so a plugin bug and a script bug must coincide to let a violation
   through.

It runs inside `npm run quality`, alongside `npm run quality:architecture`. Both print every
violation with the file and the responsible owner, never just a count.
