# 29 — Exceptions

> Authority: normative. Conflicts resolve in favor of ADRs and the security policy.

## Mandatory

- MUST cite an exception identifier matching `EXC-nnnn` — four digits, zero padded — inside every
  comment that contains `eslint-disable`. The rule reads the comment text, so the reference may sit
  after the `--` separator where ESLint already expects a justification.
- MUST register the identifier in [`../docs/exceptions/README.md`](../docs/exceptions/README.md)
  _before_ the code lands, with all of: rule, scope, owner, date, reason, alternatives considered,
  risk, tests, removal condition, and review date.
- MUST claim the next free number in sequence; identifiers are permanent and never reused, even
  after the exception is removed.
- MUST keep the suppression as narrow as the tool allows — one rule name, one file, ideally one
  function — and MUST re-enable immediately afterwards with a matching `eslint-enable`.
- MUST try relocation first: nearly every guardrail failure is a file in the wrong place, and moving
  it is cheaper than owning an exception forever.
- MUST re-justify at the review date, and MUST delete the entry and the comment together when the
  constraint that forced it is gone.

## Forbidden

- NEVER write a bare `eslint-disable` or `eslint-disable-next-line` with no rule name — an abusive
  disable is rejected on its own, before the `EXC` rule even runs.
- NEVER disable a rule for a whole file or directory, and never add an architecture rule to an
  ignore list.
- NEVER invent an identifier that has no register entry: the number is a pointer, and a dangling
  pointer is worse than no comment.
- NEVER use an exception to skip a coverage threshold because a file is "hard to test" — that is the
  file telling you it has too many responsibilities.

## The exceptions that actually exist

There is exactly one active `EXC-nnnn` in this repository:

- **EXC-0001** — `no-console`, scoped to `createConsoleSink` in
  [`../src/packages/logger/logger.factory.ts`](../src/packages/logger/logger.factory.ts). Something
  must own the console; the logger package is that owner, and it sanitizes fields before writing.
  Full record — alternatives, risk, removal condition — lives in the register.

One further carve-out exists but is deliberately **not** an `EXC`: `public/mockServiceWorker.js` is
listed in the `ignores` block of `eslint.config.mjs`. It is generated by MSW, ships with its own
blanket `/* eslint-disable */`, and is never parsed by ESLint — so there is no suppression of ours
to document. Generated vendor artifacts are excluded at the config level; hand-written code is not.

## Rationale

An undocumented suppression is a decision with no author, no expiry, and no way to tell a real
constraint from a bad afternoon. Forcing a register entry makes the cost of an exception visible —
you must name what you rejected — which is usually enough to make relocating the code the cheaper
option. One identifier in a codebase this size is the point: the number should stay boring.

## Valid

```ts
// src/packages/logger/logger.factory.ts
/* eslint-disable no-console -- EXC-0001: the logger package is the single console owner. */
const writer = level === LOG_LEVEL.Error ? console.error : console.info;
/* eslint-enable no-console */
```

## Invalid

```ts
// src/modules/auth/services/login.service.ts
/* eslint-disable */ // abusive: no rule, no reference, no scope, never re-enabled
// eslint-disable-next-line no-console -- temporary, will remove
console.log('login attempt', credentials);
```

## Enforcement

| Mechanism                                     | Command                |
| --------------------------------------------- | ---------------------- |
| `architecture/no-undocumented-eslint-disable` | `npm run lint`         |
| `unicorn/no-abusive-eslint-disable`           | `npm run lint`         |
| Register links resolve to the cited files     | `npm run quality:docs` |

Manual review where mechanical enforcement is impossible: the rule matches `EXC-\d{4}`, so
`EXC-9999` satisfies it whether or not the register has ever heard of it. The link from comment to
register entry is a human obligation — check that the number exists and that its reason still holds.

## Definition of done

- [ ] Every disable in the diff names one rule, cites a real `EXC-nnnn`, and re-enables after.
- [ ] The register entry exists with alternatives rejected, a removal condition, and a review date.
- [ ] Relocating the code was tried first, and the change says why it did not work.
