# Skill: Verify the native builds

**Use when:** anything native changes — a plugin, config, identity, or before a release.

## Required reading

- [rules/26 — Native release readiness](../rules/26-native-release-readiness.md) — sync drift and
  honest reporting.
- [check-capacitor-sync.mjs](../scripts/native/check-capacitor-sync.mjs) — what "drift" means here.
- [verify-ios-environment.mjs](../scripts/native/verify-ios-environment.mjs) — read the top comment;
  it explains the UNVERIFIED outcome.
- [capacitor.config.ts](../capacitor.config.ts) — `webDir`, `androidScheme`, and the `server.url`
  ban.
- [context/native-capability-map](../context/native-capability-map.md) — what native surface exists.

## Preconditions

- [ ] `npm run build` succeeds — `cap sync` copies `dist/`, so a stale build syncs stale assets.
- [ ] Java is installed for Gradle; `run-gradle.mjs` probes for it and fails with a real message.
- [ ] You know your platform's ceiling: **iOS cannot be compiled off macOS.** Plan to report that,
      not to hide it.

## Files

```text
android/          committed native project — regenerated by cap sync
ios/              committed native project — regenerated by cap sync
capacitor.config.ts
src/shared/config/app-identity.constants.ts   the single source of appId/appName
```

## Steps

1. **Sync first:** `npm run cap:sync` — it runs `npm run build` then `cap sync`, copying the web
   build and updating native plugin registrations.
2. **Check for drift:** `npm run cap:sync:check`. It asserts `android/app/build.gradle` and
   `ios/App/App/Info.plist` exist, that the native `appId` matches `APP_IDENTITY.appId` read out of
   `app-identity.constants.ts`, and that the tracked native tree is clean **after** a sync. A dirty
   tree means someone committed without syncing — commit the sync result.
3. **Android:** `npm run android:verify` — sync, then `android:lint`, `android:test`, and
   `android:build` (`assembleDebug`) through `scripts/native/run-gradle.mjs`. Run the pieces
   individually while iterating; run the whole thing before claiming it passes.
4. **iOS:** `npm run ios:verify`. On macOS with Xcode it runs a real build probe. **Anywhere else it
   prints `iOS build verification UNVERIFIED on this platform` and exits 0** — deliberately, because
   skipping honestly beats faking a pass. Exit 0 is not a pass; read the output.
5. **Report iOS precisely.** "iOS UNVERIFIED — requires macOS with Xcode; the committed `ios/`
   project passed structural checks only." Never write "native verified" from a Windows or Linux
   machine. See [final-validation](final-validation.md).
6. **Never commit `server.url`.** The config comment spells out the consequence: a live-reload URL
   would ship a production app that loads its code from a remote host. `androidScheme` stays `https`
   and `cleartext` stays `false`.
7. **Exercise on a device** what no gate can see: deep-link cold start ([deep-link](deep-link.md)),
   the hardware back button, permission prompts ([permission-flow](permission-flow.md)), keyboard
   resize, splash and status-bar behavior, and secure storage actually surviving a restart.
8. Review the generated native diff after adding a plugin — manifest entries and Info.plist keys
   arrive silently. Read them before committing.

## Tests

- Vitest covers the TypeScript side only: owner facades and platform facades with the plugin mocked.
  It proves your wiring, never the OS.
- `npm run android:test` runs the Android unit tests inside the native project.
- There is no automated native E2E in this repo. Device checks are manual and must be reported as
  manual — a checklist in the PR, not a green tick borrowed from another suite.
- Run the TypeScript side with
  `npx vitest run --project unit src/platform src/packages/capacitor-app`.

## Security / accessibility / native considerations

- `cleartext: false` and `androidScheme: 'https'` are security posture; a change to either needs a
  security review — see [security-review](security-review.md).
- Secure storage is only hardware-backed on device. A green web suite says nothing about the
  keychain — see [secure-storage](secure-storage.md).
- Screen readers differ from axe: verify TalkBack and VoiceOver by hand on a real screen.

## Validation

```bash
npm run cap:sync
npm run cap:sync:check
npm run android:verify
npm run ios:verify
npm run validate:native
```

## Forbidden shortcuts

- Reporting `validate:native` as fully green off macOS — `ios:verify` exits 0 while printing
  UNVERIFIED; quoting the exit code as a pass is dishonest reporting.
- Editing `android/` or `ios/` by hand instead of the config — the next `cap sync` erases it and
  `cap:sync:check` fails.
- Committing native changes without a sync — drift ships as a broken build.
- Adding `server.url` for a demo — it survives into the release nobody re-checked.
- Assuming the web build proves the native build — different bundles, different plugins, different
  permissions.

## Definition of done

- [ ] `cap:sync:check` is clean and the synced native tree is committed.
- [ ] `android:verify` passes end to end.
- [ ] iOS status is stated exactly as the script reports it, including UNVERIFIED off macOS.
- [ ] Device-only behaviors were exercised by hand and listed as manual checks.
- [ ] No `server.url`; `cleartext` is still `false`.
